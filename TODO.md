# TODO â€” Known Issues

Global audit findings (multi-agent review, 2026-07-08). All items verified against the code with file:line references. Release chores (version bumps, tag re-pinning) intentionally excluded.

## 1. User-triggerable panics â€” a library must not abort the host process

- [ ] **User-supplied `Host` header is reused as the TLS server name and panics on valid values.** `ConnectionParams::get_server_name()` prefers `host_header` verbatim ([connection_data.rs:22](src/http_clients_cache/connection_data.rs#L22)), and `HttpsConnector::connect` does `ServerName::try_from(...).unwrap()` ([https_connector.rs:73](src/http_connectors/https_connector.rs#L73)). RFC-valid `Host: api.internal:8443` (colon) or any non-ASCII host panics at connect time. Even without a port, overriding `Host` silently changes SNI **and certificate validation** â€” an undocumented cross-layer coupling. Fix: strip/parse the port, use the endpoint host for SNI unless an explicit override is given, and return `FlUrlError` (the purpose-built `HttpsInvalidDomainName` variant is never constructed) instead of unwrapping.
- [ ] **Mixing `FlUrlMode` values against one host panics.** The connection cache keys by `host:port` only ([utils.rs:3-5](src/http_clients_cache/utils.rs#L3-L5)) while the wrapper variant is frozen from `params.mode` at creation ([creators/http.rs:23-36](src/http_clients_cache/creators/http.rs#L23-L36)). A cached wrapper of another mode makes `unwrap_as_hyper` / `unwrap_as_my_http_client_request` panic ([compiled_http_request.rs:31,39](src/compiled_http_request.rs#L31)). Cached-H2 + default-mode request doesn't panic but sends a URI without scheme/authority. Fix: include the mode in the cache key or validate on checkout.
- [ ] **Scheme/platform panics inside a `Result`-returning pipeline.** `execute()` panics on `ws://`/`wss://` ([fl_url.rs:341-347](src/fl_url.rs#L341-L347)) and on unix sockets on non-unix ([fl_url.rs:393-395](src/fl_url.rs#L393-L395)); `try_new()` panics on ssh-style strings when `with-ssh` is compiled out ([fl_url.rs:121](src/fl_url.rs#L121)). All should return `FlUrlError`.
- [ ] **`try_new` panics on empty and non-ASCII URLs.** Empty string panics in url-utils `UrlBuilder::new` (`chars().next().unwrap()`); non-ASCII URLs panic on non-char-boundary byte slicing or silently corrupt host/path (char positions vs byte offsets in `UrlBuilderInner`/`RemoteEndpointInner`). Entry point: [fl_url.rs:97](src/fl_url.rs#L97). Fix belongs partly in the url-utils dependency.
- [ ] **Reading a header after reading the body panics "Body is already disposed"** â€” even though the materialized `Body` variant holds the full `HeaderMap` next to the panic site ([response_body.rs:54-56,78-80](src/response_body.rs#L54-L56); rewrite happens at [response_body.rs:156-161](src/response_body.rs#L156-L161)). `get_body_as_str()` then `get_header("Content-Type")` is an ordinary flow that currently aborts. Fix: match `Self::Body { headers, .. }` and read them.
- [ ] **Streamed body: mid-stream transport error panics the caller.** `FlResponseAsStream::get_next_chunk` does `frame.unwrap()` ([fl_response_as_stream.rs:24](src/fl_response_as_stream.rs#L24)); hyper's `Incoming` yields `Some(Err(_))` on peer reset / RST_STREAM. Also: HTTP/2 trailer frames are misreported as `ReadingHyperBodyError` instead of clean EOF, and in Http1NoHyper mode a mid-body failure surfaces as silent truncation (`Ok(None)`) because the chunk channel never sends `Err`.
- [ ] **Cancel-safety: `convert_to_slice_if_needed` takes the response out of the enum before awaiting** ([response_body.rs:145](src/response_body.rs#L145)); a body-read error or a cancelled future leaves `Hyper(None)` and every subsequent accessor panics on `unwrap`.
- [ ] **`FlUrlBody::as_json` uses `.expect()` on serialization** ([fl_url_body.rs:26](src/body/fl_url_body.rs#L26)) â€” a failing `Serialize` impl panics instead of returning an error.
- [ ] **H2 URI build unwraps on unescaped paths.** `compile_hyper_request` does `.build().unwrap()` ([fl_url.rs:526](src/fl_url.rs#L526)); since path segments are never percent-encoded, a space or non-ASCII char in a segment panics in H2 mode while the equivalent H1 error is handled gracefully ([fl_url.rs:557-568](src/fl_url.rs#L557-L568)).
- [ ] **SSH connector panics "Port is not defined"** when the tunneled target URL omits an explicit port ([ssh_connector.rs:17](src/http_connectors/ssh_connector.rs#L17)): `RemoteEndpoint::get_port()` ignores the scheme default and the `set_default_port(80)` applied earlier.

## 2. Connection pool / lifecycle (architectural)

- [ ] **Double-checkout: connections are shared, not checked out.** `get_connection` inserts the freshly created wrapper into the pool Vec AND returns it ([http_connections_cache.rs:218-226](src/http_clients_cache/http_connections_cache.rs#L218-L226)), and every `put_connection_back` pushes another clone of the same `Arc` â€” the pool fills with duplicates of one physical connection and concurrent tasks routinely share the same HTTP/1 client.
- [ ] **Connection returned to the pool at response-HEAD time, before the body is consumed** ([fl_url.rs:809-813](src/fl_url.rs#L809-L813)). Bodies are streaming (all bodies in Http1Hyper/H2, chunked in Http1NoHyper) and `FlUrlResponse` holds no reference to the connection â€” a concurrent request on the same pooled connection stalls behind the in-flight body, times out, and its disconnect tears down the first request's still-streaming body.
- [ ] **No pool eviction on failure or `Connection: close`.** The error arm of `execute_with_retry` neither disposes nor removes the connection; `FlUrlResponse::drop_connection()` and the exported `DropConnectionScenario` trait ([fl_drop_connection_scenario.rs:3-19](src/fl_drop_connection_scenario.rs#L3-L19)) are dead code with zero call sites â€” connections the server announced it will close are pooled anyway.
- [ ] **`do_not_reuse_connection` tears the socket down under a streaming body** â€” the creator path's `put_connection_back` is a no-op ([creators/http.rs:50-54](src/http_clients_cache/creators/http.rs#L50-L54)), so the last `Arc` drops right after the response head and `Drop` shuts the socket down while the user is still reading. Same tear on the pool-full silent drop ([http_connections_cache.rs:245-246](src/http_clients_cache/http_connections_cache.rs#L245-L246)). The flag is also silently ignored for ssh:// targets ([fl_url.rs:334,438-445](src/fl_url.rs#L334)).
- [ ] **`max_connections` is hardcoded to 5, private, and counts Vec entries** rather than distinct connections ([http_connections_cache.rs:35](src/http_clients_cache/http_connections_cache.rs#L35)) â€” with the duplicate-Arc bug the 5 slots can all be one physical connection.
- [ ] **Process-global `CLIENTS_CACHED` grows without bound** ([lib.rs:47](src/lib.rs#L47)): stale eviction runs only on checkout of the same key; hosts never contacted again leak their entries; no sweep, no clear/shutdown API.
- [ ] **TLS identity is not part of the HTTPS cache key.** SNI (from the Host header) and the client certificate are baked into the connector at creation ([creators/https.rs:21-28](src/http_clients_cache/creators/https.rs#L21-L28)) but the key is `host:port` â€” later requests with a different Host header or different/no client cert silently reuse a connection handshaken with the first request's identity.
- [ ] **`set_not_used_connection_timeout` is a no-op** ([fl_url.rs:158](src/fl_url.rs#L158)): the field is stored but never read; the TTL actually applied is the separate `reuse_connection_timeout_sec = 120`, which has no builder method.
- [ ] **Retry-on-retry without an idempotency gate.** `execute_with_retry` ([fl_url.rs:804-825](src/fl_url.rs#L804-L825)) replays the same compiled request on ANY error when `with_retries > 0`, overriding the dependency's deliberate refusal to replay non-idempotent requests on timeout â€” a POST can execute server-side up to `max_retries+1` times, multiplied by up to 4 dependency-internal attempts on disconnects.

## 3. HTTP protocol correctness

- [ ] **Http1NoHyper body framing ignores request method and status** ([fl_url.rs:632](src/fl_url.rs#L632) + dependency read loop): `head()` hangs until timeout on any response carrying `Content-Length` (body never arrives); responses with neither `Content-Length` nor chunked encoding (HTTP/1.0-style close-delimited) are silently returned as empty â€” data loss with status 200.
- [ ] **Auto-generated `Host` header omits the port** (uses `get_host()` instead of `get_host_port()`, [fl_url.rs:543](src/fl_url.rs#L543) hyper / [fl_url.rs:598](src/fl_url.rs#L598) non-hyper) â€” violates RFC 9110 Â§7.2 for non-default ports.
- [ ] **Query without path produces request target `?a=b` with no leading `/`** ([fl_url.rs:511](src/fl_url.rs#L511)): default mode fails with a misleading error, Http1NoHyper sends a malformed request line.
- [ ] **`append_path_segment` performs no percent-encoding** ([fl_url.rs:289](src/fl_url.rs#L289)) while `append_query_param` does â€” reserved/whitespace chars cause mode-dependent errors, a panic (H2), or silent URL corruption.
- [ ] **Duplicate headers on the wire:** body-derived `Content-Type` ([fl_url.rs:497,579](src/fl_url.rs#L497)) and compression's `Content-Encoding` ([fl_url.rs:461](src/fl_url.rs#L461)) are appended, never replaced â€” a user-set header of the same name yields duplicated singleton headers.
- [ ] **SSH tunnel silently downgrades `->https://` targets to plaintext HTTP** and hardcodes port 80: `execute()` short-circuits into `execute_ssh` before the scheme match ([fl_url.rs:333-338,438-445](src/fl_url.rs#L333-L338)).
- [ ] **WebSocket upgrade is a dead end in all three wrapper arms** ([my_http_client_wrapper_inner.rs:29-42](src/my_http_client_wrapper/my_http_client_wrapper_inner.rs#L29-L42)): Hyper arm drops the established socket and re-caches the now HTTP-dead connection; Http1NoHyper reaches `TConnector::reunite` which is `panic!(...)` in every connector ([http_connector.rs:42](src/http_connectors/http_connector.rs#L42) et al.); http1 arm loses the stream in `into_response()` ([fl_response.rs:35](src/fl_response.rs#L35)). Normalize to an explicit error and never re-pool such a connection.
- [ ] **Documented unix-socket syntax doesn't work.** `unix:///path` (README:601) panics with `InvalidUrl` at construction ([fl_url.rs:82](src/fl_url.rs#L82)); the only working form is the undocumented `http+unix://...`. Fix parser or docs.

## 4. API surface / error handling

- [ ] **`FlUrlError` implements neither `Display` nor `std::error::Error`** ([fl_url_error.rs:30](src/errors/fl_url_error.rs#L30), same for `FlUrlReadingHeaderError`) â€” doesn't interoperate with `?`, `Box<dyn Error>`, or anyhow. Also: variants `Timeout`, `HyperError`, `IoError`, `HttpsInvalidDomainName`, `InvalidHttp1HandShake`, `CanNotEstablishConnection`, `SshSessionError` are never constructed; every runtime failure funnels into `MyHttpClientError` ([fl_url.rs:819](src/fl_url.rs#L819)), so timeouts are indistinguishable via the advertised taxonomy.
- [ ] **`accept_invalid_certificate()` is a silent no-op** ([fl_url.rs:284-287](src/fl_url.rs#L284-L287)): the flag is never copied into `ConnectionParams` and `HttpsConnector` builds TLS config only from the client certificate â€” validation is always strict while the API promises otherwise.
- [ ] **No timeout covers body reads.** `request_timeout` bounds only `do_request` (up to response headers); body collection ([response_body.rs:152](src/response_body.rs#L152)) and streaming reads have no timeout at all.
- [ ] **No size cap on body buffering.** All non-streaming reads buffer the whole response into memory with no limit or Content-Length validation; in Http1NoHyper even `get_body_as_stream` can't bound memory (length-based bodies are fully buffered by the dependency before `do_request` returns).
- [ ] **No response decompression** ([response_body.rs:169](src/response_body.rs#L169)): `Content-Encoding` is never inspected, so gzip bodies reach `get_json`/`get_body_as_str` as raw compressed bytes â€” asymmetric with request-side compression ([fl_url.rs:461](src/fl_url.rs#L461)).
- [ ] **No redirect handling anywhere** (3xx returned raw) and this is documented nowhere â€” surprising for a client modeled on Flurl.NET, which follows redirects by default. Document or implement.
- [ ] **Public API shape varies by platform/feature without `#[non_exhaustive]`**: cfg-gated `FlUrlError::SshSessionError` variant ([fl_url_error.rs:17](src/errors/fl_url_error.rs#L17)) and gated pub field `ConnectionParams::ssh_session` ([connection_data.rs:16-17](src/http_clients_cache/connection_data.rs#L16-L17)).
- [ ] **Windows default-feature build is never CI-exercised** while both `ring` and `aws-lc-sys` end up in the graph via my-tls â†’ rustls/rcgen ([Cargo.toml:40](Cargo.toml#L40)); CI covers only `--no-default-features` on Windows.
